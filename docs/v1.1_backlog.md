## Risk Analysis

### `created_at` fallback: read-only vs. write-back

**Risk of read-only**: Sort order is inconsistent — packages without `created_at` sort as "" and appear last.

**Risk of write-back**: Modifying 100+ existing package JSONs is a data migration; needs backup + rollback plan.

**Recommendation for Slice B**: Start with API-layer `ctime` proxy sort; write-back only if manually triggered (not automatic on every read).

---

### `has_package` semantics creep

**Current**: `has_package` = `note_package.json` file exists on disk.

**Potential future ask**: `has_package` = content is valid (pages array non-empty, images exist).

**Risk**: If semantics change, replay/export guard conditions break.

**Recommendation**: Document `has_package = file-exists` explicitly; any content-validity check belongs in a separate field `has_valid_content` or similar.

---

### Pagination vs. bulk delete select-all

**Current**: Bulk delete selects from full filtered list (no pagination).

**After pagination**: Select-all could mean "all on current page" or "all filtered results across pages".

**Decision required in Slice C**: Propose default = current page only (safer); cross-page selection deferred.
